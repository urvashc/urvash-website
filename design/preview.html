Mavrck / Later · 2022–2023
Senior Product Manager · Data Platform

Centralising ML deployment across a fragmented data platform

1 qtrfull migration
6+ML models unified
↓cloud spend
// context

Mavrck acquired Later in 2022, creating a combined influencer marketing platform with a significant technical debt problem: two separate data stacks, three ML deployment pipelines, no shared infrastructure. Models were deployed wherever the team that built them happened to work — Kubernetes here, Lambda there, a SageMaker endpoint that one engineer understood.

I was brought in to own the data platform roadmap after the merger. The ML deployment problem wasn't the only issue, but it was the one compounding fastest. Cloud costs were climbing. Model monitoring was non-existent. Every new model created another one-off deployment pattern.

"The problem wasn't that we had six models. It was that we had six deployment philosophies."
// the problem

Before writing a single line of spec, I needed to understand the actual state of the infrastructure. I spent the first two weeks reading schemas, reviewing deployment configs, and sitting with the ML engineers. Not to tell them what to build — to understand what we actually had.

Three deployment patterns across two clouds, no shared feature store, no centralized monitoring, and model metadata scattered across Confluence pages and Slack threads. The technical debt was real but not insurmountable. The organizational debt was harder.

// the approach

Phased consolidation: first, a unified deployment surface (one pipeline, two environments); second, a shared feature registry; third, one monitoring layer with standardized alerting. The spec went through five revisions — not because the technical direction changed, but because each revision forced a harder conversation about ownership.

// outcome

Six models migrated in one quarter. Cloud spend down. New models now have a clear deployment path. The monitoring layer that didn't exist now pages on-call when a model's output distribution drifts.

The boring outcome: a table in Confluence that says exactly which team owns each model and what the SLA is. That table didn't exist before. It matters more than any architecture diagram.
